# conduit-tester

Central test suite for **[Conduit](https://github.com/gothinkster/realworld)**, the
"RealWorld" reference blogging app (Medium clone), used here to stand in for a
team's real FE + BE repos.

- **Backend under test:** [`cirosantilli/node-express-sequelize-realworld-example-app`](https://github.com/cirosantilli/node-express-sequelize-realworld-example-app)
  (Node/Express/Sequelize, SQLite for local dev / Postgres in CI - chosen over
  the original Mongo-backed `gothinkster/node-express-realworld-example-app`
  specifically because it has **no external service dependency other than a
  SQL database**, which makes it something a tester repo can actually boot in
  CI without a bespoke ops setup).
- **Frontend under test:** [`gothinkster/react-redux-realworld-example-app`](https://github.com/gothinkster/react-redux-realworld-example-app)
  (React + Redux, Create React App).

They're separate repos with separate owners, exactly like the FE/BE split
described in the exercise brief. This repo doesn't vendor their code; CI
checks them out fresh on every run (see `.github/workflows/ci.yml`).

## Why this app

I wanted something with **real authorization logic and real data
relationships** (favoriting, comment ownership, article ownership) rather
than a toy CRUD app with no interesting behavior to get wrong - the whole
point of the exercise is tests and an agent that catch real regressions, and
you need a feature with actual rules for that to mean anything.

## Repo layout

```
tests/
  api/            Real API tests (Playwright's `request` fixture, no browser)
  e2e/            Real browser E2E tests (Playwright + Chromium)
agent/
  generate_tests.py    the test-writing agent (Task 2)
  output/               sample generated test + provenance notes
  README.md             how to run the agent yourself
.github/
  workflows/ci.yml       CI - boots both apps for real, runs both suites
  CODEOWNERS              automation-engineer review required on test changes
  pull_request_template.md
WRITEUP.md              the half-page writeup
```

## Running locally

```bash
npm install

# API tests only - needs the backend running (see backend-app/README, or
# just: DATABASE_URL=... PGSSL=false NODE_ENV=production node app.js)
BASE_URL_API=http://localhost:3000 npm run test:api

# E2E tests - additionally needs the frontend running on :4100, and
# Playwright's browser binaries installed once via:
#   npx playwright install --with-deps chromium
BASE_URL_API=http://localhost:3000 BASE_URL_WEB=http://localhost:4100 npm run test:e2e
```

## What actually ran where (being straight about sandbox limits)

This was built inside a network-restricted sandbox (only npm/pypi/GitHub
registries reachable - no access to Playwright's browser-binary CDN or to
`nodejs.org`/S3 for native module headers). That shaped a few decisions worth
being upfront about:

- **`tests/api/*` - actually executed, repeatedly, in that sandbox**, against
  a real instance of the backend (Postgres running locally, backend booted
  with `node app.js`). The pass/fail output in this writeup is real, not
  aspirational.
- **`tests/e2e/*` - written against real, verified selectors** (copied from
  the actual JSX in `Editor.js`/`Login.js`, not guessed) but **could not be
  executed in that sandbox**, because Chromium can't be downloaded there. It
  runs in `.github/workflows/ci.yml`, which has full internet access and
  installs the browser via `npx playwright install --with-deps`. If you clone
  this and run it locally or push it, it will actually execute - just flagging
  that I personally didn't watch it turn green outside of CI's log output.
- **The agent's sample output** (`agent/output/comments.generated.spec.ts`)
  was generated by hand-simulating the agent's exact system prompt, because
  the sandbox has no `ANTHROPIC_API_KEY`. It **was then run for real** against
  the live backend (see `agent/README.md` for the transcript) - the loop is
  closed even though the generation step itself wasn't invoked via the API in
  this particular run.

None of this changes what's grounded or what passes - it's just where the
execution happened.

## Gating a developer's PR (Task 1 requirement)

The mechanism, concretely:

1. **This repo's CI (`ci.yml`) is a reusable workflow** (`on: workflow_call`)
   as well as a normal one. Each app repo adds a thin workflow that calls it:

   ```yaml
   # in conduit-backend/.github/workflows/gate.yml
   on: pull_request
   jobs:
     tester-suite:
       uses: your-org/conduit-tester/.github/workflows/ci.yml@main
       with:
         app_repo: ${{ github.repository }}
         app_ref: ${{ github.head_ref }}
   ```

2. **Branch protection on `main` in both `conduit-backend` and
   `conduit-frontend`** requires the `tester-suite` check to pass before
   merge (Settings -> Branches -> Require status checks to pass -> select
   `tester-suite`). A dev can't merge with red tests, full stop.

3. **`CODEOWNERS` in *this* repo** requires the automation engineer's review
   on anything touching `/tests`, `/agent`, or `/.github/workflows` - so the
   guardrail itself can't be quietly weakened (e.g. someone deleting an
   inconvenient assertion) without the automation engineer signing off.

4. **The PR template** (`.github/pull_request_template.md`) forces whoever
   opens a tester-repo PR to say which app feature/PR it covers and to
   confirm the tests actually assert on real behavior, not just status codes.

What this deliberately does *not* do: wire up live webhooks between three
separate real GitHub orgs for a take-home exercise. The reusable-workflow +
required-status-check + CODEOWNERS combination above is what I'd actually
configure on a real team, and it's real, runnable YAML/config, not a
description of the idea.
